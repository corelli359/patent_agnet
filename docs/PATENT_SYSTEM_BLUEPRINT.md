# 专利全流程产出系统总方案

## 1. 项目定位

本项目的目标不是做 6 个彼此独立的 Skill 页面，而是做一套围绕“专利项目”展开的全流程产出系统。用户从一个模糊创意进入系统，经过技术澄清、现有技术检索、交底书生成、申请文件起草、审查模拟、修复答复，最终产出一份具备较高代理交付价值的专利申请包。

系统的核心对象应当是 `PatentDraft`，但它在产品层面应升级为“专利项目容器”，统一承载：

- 项目基础信息：标题、类型、技术领域、业务背景、目标保护范围
- 中间产物：发明意图、检索快照、交底书、权利要求书、说明书、摘要
- 质量状态：阶段完成度、风险项、审查记录、修复策略
- 最终交付物：交付包、答复意见书、后续人工复核建议

## 2. 产品主流程

### 阶段 A：项目立项
- 输入：标题、技术领域、初始问题描述、业务价值
- 输出：项目卡片、阶段状态、下一步建议
- 目标：把零散创意挂到统一项目下，避免信息在多个 Skill 中断裂

### 阶段 B：发明意图沉淀
- 依托 Skill 1 多轮对话
- 输出：发明名称、技术问题、现有不足、解决方案、关键步骤、创新点、预期效果
- 质量门槛：必须能回答“解决什么问题”“为什么现有方案不够”“保护重点是什么”

### 阶段 C：现有技术检索
- 依托 Skill 6 检索 Google Patents / 其他源
- 输出：检索历史、专利结果、关键词分析、IPC 分布、建议二次检索语句
- 质量门槛：至少形成 1 组与当前项目绑定的检索分析快照

### 阶段 D：技术交底书
- 依托 Skill 2，将发明意图扩展为适合代理师继续加工的结构化材料
- 输出：技术领域、背景技术、发明内容、实施方式
- 质量门槛：实施方式必须能支撑后续权利要求的关键技术特征

### 阶段 E：申请文件起草
- 依托 Skill 3
- 输出：权利要求书、说明书、摘要
- 质量门槛：独立权利要求边界清晰，从属权利要求有层次，说明书与权利要求一致

### 阶段 F：审查模拟与修复
- 依托 Skill 4 + Skill 5
- 输出：缺陷报告、修复策略、答复意见书
- 质量门槛：至少完成一轮内部审查闭环，明确风险位置与修订动作

### 阶段 G：最终交付
- 输出：专利项目交付包
- 内容包括：项目概览、检索分析、交底书、申请文件、审查结论、修复建议、人工复核建议

## 3. 统一架构设计

### 3.1 后端
- `backend/app/services/`
  - 保留现有 6 个 Skill Service
  - 新增 `PatentWorkflowService` 作为项目编排层
- `shared/db/models.py`
  - `PatentDraft` 继续作为核心表
  - `ExaminationRecord`、`SearchHistory` 作为项目附属证据链
- `backend/app/main.py`
  - 在保留单 Skill API 的同时，新增项目级 API

### 3.2 数据主线
推荐将 `PatentDraft.content` 统一整理为以下结构：

```json
{
  "project_profile": {},
  "workflow": {},
  "invention_intent": {},
  "invention_intent_doc": "",
  "prior_art_search": {
    "searches": []
  },
  "disclosure": {},
  "disclosure_doc": "",
  "claims": "",
  "specification": "",
  "abstract": "",
  "repair_strategies": [],
  "response_opinion": "",
  "delivery_package": {}
}
```

这样可以确保所有中间结果都围绕一个项目聚合，不再散落在单一 Skill 响应里。

## 4. 当前代码现状判断

现状不是“从零开始”，而是“能力已分散实现，但缺主流程”：

- 已有基础能力：Skill 1-6 的服务层、检索分析器、数据库模型、FastAPI 接口
- 已有中间数据容器：`PatentDraft`
- 主要缺口：
  - 缺项目级编排层
  - 缺项目总览、阶段状态、质量检查
  - 缺检索结果与项目的显式绑定
  - 缺最终交付包
  - 前端仍停留在 Skill 页面，不是项目工作台

## 5. 实施路线

### 第一阶段：打通主骨架
- 新增项目工作流服务
- 新增项目创建、项目总览、挂接检索、交付包 API
- 形成统一方案文档

### 第二阶段：提升质量闭环
- 为 Skill 3 增加更强的权利要求校验
- 将 Skill 4 的缺陷结果沉淀进项目工作流
- 将 Skill 5 的修复策略和答复书自动写回项目

### 第三阶段：建设前端工作台
- 以“项目”为中心展示阶段进度，而不是 Skill 列表
- 一页串联“当前阶段、风险、交付物、下一步动作”
- 增加项目列表、历史版本、交付包预览

### 第四阶段：提升检索与论证深度
- 引入 PDF 下载与解析
- 建立本地向量库，支持相似技术比对
- 增加“本申请 vs 对比文件”的特征差异表

## 6. 质量标准

这套系统最终的目标不是“能生成文档”，而是“能生成足以进入人工精修和正式申报的高质量底稿”。建议以以下标准验收：

- 文档完整性：意图、检索、交底、权利要求、说明书、摘要必须连贯
- 可追溯性：每个关键结论都能回溯到检索记录或中间文档
- 风险显式化：新颖性、清楚性、支持性问题不能隐含，必须显式列出
- 可交付性：最终交付包可直接发给代理师或内部知识产权团队复核

## 7. 本轮已落地的方向

本轮开发优先完成“产品主线骨架”：

- 新增项目级工作流服务 `PatentWorkflowService`
- 让 `PatentDraft` 升级为项目总览入口
- 支持检索结果挂接到专利项目
- 支持生成项目交付包

这一步的价值在于，后续每增强一个 Skill，都能直接反映到整个项目工作流上，而不是继续增加孤立模块。
